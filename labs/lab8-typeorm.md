## **Теоретичні відомості**

### **Entity та Repository**

**Entity** це клас TypeScript, позначений декоратором `@Entity()`, що описує структуру таблиці в базі даних. Кожен екземпляр такого класу відповідає одному рядку. **Repository** це типізований об'єкт, який інкапсулює логіку доступу до даних конкретної сутності та надає методи для CRUD-операцій (`find`, `findOne`, `save`, `update`, `delete`).

```typescript
// src/tasks/task.entity.ts
import {
  Entity,
  PrimaryGeneratedColumn,
  Column,
  CreateDateColumn,
} from "typeorm";

@Entity("tasks")
export class Task {
  @PrimaryGeneratedColumn()
  id: number;

  @Column({ length: 100 })
  title: string;

  @Column({ type: "text", nullable: true })
  description: string | null;

  @Column({ default: "pending" })
  status: "pending" | "in-progress" | "done";

  @Column({ default: "medium" })
  priority: "low" | "medium" | "high";

  @CreateDateColumn()
  createdAt: Date;
}
```

Сутність реєструється у feature-модулі через `TypeOrmModule.forFeature([Task])`, після чого репозиторій можна інжектувати в сервіс через декоратор `@InjectRepository`:

```typescript
@Injectable()
export class TasksService {
  constructor(
    @InjectRepository(Task)
    private readonly tasksRepository: Repository<Task>,
  ) {}

  findAll(): Promise<Task[]> {
    return this.tasksRepository.find();
  }
}
```

### **Migrations**

**Migration** (міграція) це версійований опис зміни схеми бази даних. Кожна міграція містить дві операції: `up` (застосувати зміну) та `down` (відкотити). На відміну від `synchronize: true`, який автоматично перебудовує схему та може видалити дані, міграції дають контрольований і відтворюваний процес еволюції бази.

CLI TypeORM генерує міграції автоматично на основі різниці між поточними сутностями та станом БД, виконує їх та дозволяє відкочувати назад.

### **Relations: ManyToMany**

Коли кілька записів однієї таблиці можуть бути пов'язані з кількома записами іншої (наприклад, задача має кілька тегів, а тег має багато задач), використовується зв'язок `@ManyToMany`. Він реалізується через **проміжну (junction) таблицю**, яку TypeORM створює автоматично. Декоратор `@JoinTable` ставиться лише на одній стороні зв'язку та визначає її як власника.

```typescript
@ManyToMany(() => Tag, { onDelete: "CASCADE" })
@JoinTable()
tags: Tag[];
```

Опція `onDelete: 'CASCADE'` означає, що при видаленні задачі або тега відповідні записи у junction-таблиці прибираються автоматично на рівні БД. Без неї спроба видалити сутність, прив'язану до іншої через цей зв'язок, призведе до помилки порушення зовнішнього ключа.

За замовчуванням TypeORM **не завантажує** пов'язані сутності. Щоб отримати задачу разом із тегами, потрібно явно вказати `relations`:

```typescript
this.tasksRepository.findOne({
  where: { id },
  relations: { tags: true },
});
```

## **Вимоги до реалізації**

### **Передумови**

- Робота виконується на основі застосунку з лабораторної роботи №7
- Створюється новий репозиторій, до якого спочатку копіюється код попередньої лаби
- PostgreSQL запущений локально; база даних для лаби створена

### **Конфігурація**

- Підключені пакети `@nestjs/typeorm`, `typeorm`, `pg`
- Параметри підключення (хост, порт, користувач, пароль, ім'я БД) зчитуються через `ConfigService` зі змінних середовища; у репозиторії наявний `.env.example` з усіма потрібними ключами
- `TypeOrmModule.forRootAsync` використовується для асинхронної конфігурації; `synchronize: false`
- Окремий файл `data-source.ts` для CLI-команд міграцій

### **Сутності та модуль**

- Сутність `Task` з полями `id`, `title`, `description`, `status`, `priority`, `createdAt`; типи та обмеження відповідають змісту полів (довжина рядків, nullable, default-значення)
- Сутність `Tag` з полями `id` та `name` (унікальне)
- Зв'язок між `Task` та `Tag` типу `ManyToMany`; junction-таблиця створюється автоматично
- Сутності зареєстровані у відповідному модулі через `TypeOrmModule.forFeature`

### **Міграції**

- Перша міграція згенерована автоматично через CLI на основі сутностей; містить створення таблиць `tasks`, `tags` та junction-таблиці зі зовнішніми ключами
- У `package.json` додані npm-скрипти для команд `migration:generate`, `migration:run`, `migration:revert`
- Жодних SQL-запитів у коді сервісів та контролерів; уся робота з даними виконується через методи репозиторію

### **Сервіс**

- Усі методи сервісу (`findAll`, `findOne`, `findByStatus`, `create`, `update`, `remove`) виконують реальні запити до БД через репозиторій; масив у пам'яті повністю прибрано
- Сервіс не містить SQL-рядків і HTTP-виключень; повертає сутність, `null` або `boolean`, як у лабі №7
- Методи отримання задач (`findAll`, `findOne`, `findByStatus`) повертають задачі разом із пов'язаними тегами через опцію `relations`
- Створення задачі підтримує необов'язковий масив `tagIds`; якщо переданий, задача зберігається разом із прив'язкою до існуючих тегів

### **Ендпоінти**

Усі ендпоінти з лабораторної роботи №7 продовжують працювати з тими ж URL та HTTP-статусами; у відповідях `GET /tasks`, `GET /tasks/:id` та `GET /tasks/search` тепер міститься масив пов'язаних тегів. Додатково з'являються ендпоінти для роботи з тегами:

| Метод  | URL       | Статус                                |
| ------ | --------- | ------------------------------------- |
| POST   | /tags     | 201 / 400                             |
| GET    | /tags     | 200 (порожній масив якщо тегів немає) |
| PATCH  | /tags/:id | 200 / 400 / 404                       |
| DELETE | /tags/:id | 204 / 404                             |

`POST /tasks` приймає необов'язкове поле `tagIds: number[]` для прив'язки до існуючих тегів при створенні. Видалення тега або задачі автоматично прибирає відповідні записи у junction-таблиці через каскадне видалення на рівні БД (опція `onDelete: 'CASCADE'` у декораторі зв'язку). Валідація через `class-validator` зберігається.

### **Тестування**

Перевірте за допомогою curl, Postman або Bruno:

- усі ендпоінти CRUD з попередньої лаби продовжують працювати з реальною БД
- після перезапуску застосунку дані зберігаються
- створення задачі з масивом `tagIds` коректно записує зв'язки в junction-таблицю
- запити `GET /tasks` та `GET /tasks/:id` повертають задачі з масивом пов'язаних тегів
- запит до неіснуючого тега у `tagIds` обробляється коректно (на ваш вибір: 404 або ігнорування неіснуючих)
- видалення тега, прив'язаного до задач, не призводить до помилки; після видалення задачі зберігаються, а їхні зв'язки з цим тегом прибираються

Додатково перевірте через SQL-клієнт або `psql`, що структура таблиць відповідає сутностям, а junction-таблиця заповнюється при створенні зв'язків та очищається при видаленні тега.
