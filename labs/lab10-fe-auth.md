# **Лабораторна робота №10**

**Тема:** Аутентифікація у React-застосунку

**Мета:** Реалізувати frontend-частину для серверного застосунку з аутентифікацією (з ЛР 9): форми реєстрації та логіну, збереження JWT-токена, автоматичне додавання токена до запитів, захищені маршрути та відображення інформації про поточного користувача. Освоїти інтеграцію React-застосунку з REST API, що використовує JWT.

**Обладнання:** IBM-сумісні ПК

**Програмне забезпечення:** ОС Windows 10/11, macOS або Linux; Node.js (v18+); npm; веб-браузер з React Developer Tools; VS Code; обліковий запис GitHub; Git; запущений бекенд з ЛР 9

---

## **Контрольні запитання**

1. Як клієнт передає JWT-токен серверу при кожному запиті та яку роль виконує заголовок `Authorization: Bearer <token>`?
2. У чому різниця між зберіганням токена у `localStorage` та `sessionStorage`? Які переваги та недоліки кожного підходу з точки зору безпеки та користувацького досвіду?
3. Що таке захищений маршрут (protected route) і як він реалізується у React Router?
4. Як працює інтерсептор у HTTP-клієнті `axios` і навіщо він потрібен у застосунку з аутентифікацією?
5. Чому не варто декодувати JWT-токен на клієнті для перевірки прав доступу? Що клієнт може робити з payload токена і яких дій робити не повинен?

## **Теоретичні відомості**

### **Передача токена з клієнта на сервер**

Після успішного логіну сервер повертає JWT-токен у тілі відповіді. Клієнт зберігає цей токен та додає його до заголовка `Authorization` у форматі `Bearer <token>` при кожному запиті до захищених ендпоінтів. Якщо токен відсутній, прострочений або має невалідний підпис, сервер повертає `401 Unauthorized`.

```http
GET /auth/me HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

### **Зберігання токена на клієнті**

Поширеними варіантами зберігання є `localStorage` та `sessionStorage`. `localStorage` зберігає дані між сесіями браузера, `sessionStorage` — лише до закриття вкладки. Обидва підходи доступні JavaScript-коду на сторінці, що робить їх вразливими до XSS-атак. Для навчального проєкту допустимим є `localStorage`; у промислових застосунках для критичних систем розглядають httpOnly cookies.

### **HTTP-інтерсептори**

Інтерсептор — функція, що автоматично виконується перед кожним запитом або після кожної відповіді HTTP-клієнта. У контексті аутентифікації інтерсептор запиту додає заголовок `Authorization` з токеном до всіх запитів, а інтерсептор відповіді обробляє статус `401`, перенаправляючи користувача на сторінку логіну при простроченні токена.

### **Захищені маршрути**

Захищений маршрут — компонент-обгортка, що перевіряє стан аутентифікації перед рендером дочірнього маршруту. Якщо користувач не аутентифікований, виконується редирект на сторінку логіну. У React Router це зазвичай реалізується через `<Outlet>` всередині компонента-охоронця або через `<Navigate>` для перенаправлення.

### **Глобальний стан аутентифікації**

Стан аутентифікації (поточний користувач, токен, статус завантаження) потрібен у багатьох частинах застосунку: у формах, у header, у захищених маршрутах, у HTTP-клієнті. Для уникнення prop drilling та узгодженості стану використовують глобальне сховище — React Context або зовнішню бібліотеку (наприклад, Zustand).

## **Завдання**

Реалізувати React-застосунок, що інтегрується з бекендом з ЛР 9. Застосунок повинен підтримувати реєстрацію, логін, відображення інформації про поточного користувача та логаут.

### **Сторінки застосунку**

| Маршрут     | Доступ                  | Призначення                                                                  |
| ----------- | ----------------------- | ---------------------------------------------------------------------------- |
| `/login`    | Лише неаутентифікованим | Форма входу (email + пароль)                                                 |
| `/register` | Лише неаутентифікованим | Форма реєстрації (email + пароль)                                            |
| `/profile`  | Лише аутентифікованим   | Сторінка з інформацією про поточного користувача (email, id, дата створення) |

### **Очікувана поведінка**

- Після успішної реєстрації користувач отримує повідомлення про створення акаунта та перенаправляється на сторінку логіну
- Після успішного логіну токен зберігається на клієнті, користувач перенаправляється на `/profile`
- На сторінці `/profile` відображається інформація про користувача, отримана через `GET /auth/me`
- При спробі відкрити `/profile` без аутентифікації відбувається редирект на `/login`
- При спробі відкрити `/login` або `/register` будучи вже аутентифікованим відбувається редирект на `/profile`
- Header відображає різний зміст залежно від стану аутентифікації: email користувача та кнопку «Вийти» для аутентифікованих, посилання «Увійти» та «Зареєструватися» для решти
- При натисканні «Вийти» токен видаляється, користувач перенаправляється на `/login`
- Помилки сервера (`400`, `401`, `409`) відображаються у формах у зрозумілому вигляді для користувача
- Після перезавантаження сторінки стан аутентифікації зберігається — токен зчитується зі сховища, дані користувача завантажуються через `GET /auth/me`

### **Архітектурні вимоги**

- Глобальний стан аутентифікації реалізований через React Context або Zustand. Сховище містить щонайменше: поточного користувача, статус завантаження та функції `login`, `register`, `logout`
- HTTP-клієнт реалізований через `axios` з налаштованими інтерсепторами: інтерсептор запиту автоматично додає заголовок `Authorization` до всіх запитів за наявності токена; інтерсептор відповіді обробляє статус `401`, очищуючи стан аутентифікації при простроченні токена
- Захищені маршрути реалізовані через окремий компонент-обгортку, що використовує стан аутентифікації для прийняття рішення про рендер або редирект
- Форми реалізовані через React Hook Form з валідацією через Zod. Правила валідації узгоджені з backend-валідацією (мінімальна довжина пароля, формат email)
- Запити до API виконуються через TanStack Query: мутації для `login` та `register`, запит для `GET /auth/me`. Логаут реалізується локально — видаленням токена зі сховища та очищенням стану, без HTTP-запиту до сервера
- URL бекенду читається зі змінної середовища `VITE_API_URL` (файл `.env`)

### **Обробка станів завантаження та помилок**

- Кнопки сабміту форм блокуються під час виконання запиту, відображаючи стан завантаження
- Помилки з сервера відображаються в інтерфейсі: `401` при логіні — «невірний email або пароль», `409` при реєстрації — «користувач з таким email вже існує», `400` — повідомлення про невалідні дані
- На сторінці `/profile` під час початкового завантаження даних користувача відображається індикатор завантаження

### **CORS**

Для роботи frontend-застосунку з бекендом необхідно увімкнути CORS на стороні backend. У `main.ts` Nest.js застосунку (з ЛР 9) додайте:

```typescript
app.enableCors({
  origin: "http://localhost:5173",
});
```

> Це єдина зміна, яка потрібна в коді бекенду з ЛР 9.

## **Типові помилки та їх вирішення**

| Проблема                                                                  | Рішення                                                                                                                                                                         |
| ------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CORS policy: No 'Access-Control-Allow-Origin' header` у консолі браузера | Увімкніть CORS на бекенді через `app.enableCors({ origin: 'http://localhost:5173' })`. Перевірте, що origin збігається з адресою, на якій працює dev-сервер frontend            |
| Захищений маршрут редиректить навіть для аутентифікованого користувача    | Початковий стан сховища повинен мати статус «завантаження», поки не виконається перший запит `GET /auth/me`. Рендерити редирект лише після завершення цього запиту              |
| Після перезавантаження сторінки користувач «вилогінюється»                | Переконайтесь, що при ініціалізації застосунку відбувається спроба завантажити користувача через `GET /auth/me`, якщо в `localStorage` присутній токен                          |
| Заголовок `Authorization` не додається до запитів                         | Перевірте, що інтерсептор `axios` зареєстрований до того, як виконується перший запит. Інтерсептор повинен зчитувати токен зі сховища при кожному запиті                        |
| Сторінка логіну/реєстрації відображається аутентифікованому користувачу   | Додайте компонент-обгортку, що виконує редирект на `/profile`, якщо користувач вже аутентифікований                                                                             |
| Форма не показує помилку від сервера                                      | Помилки HTTP-запиту приходять окремо від помилок валідації Zod. Зберігайте їх в окремому стані компонента або через TanStack Query (`mutation.error`) і відображайте над формою |
