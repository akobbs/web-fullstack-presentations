## **Теоретичні відомості**

### **Аутентифікація vs авторизація**

**Аутентифікація** (authentication) — процес перевірки, _хто_ робить запит. Реалізується через перевірку облікових даних (email + пароль) або токена.

**Авторизація** (authorization) — процес перевірки, чи має ідентифікований користувач _право_ виконати конкретну дію. Реалізується через ролі, права доступу або перевірку власника ресурсу.

У типовому потоці клієнт спочатку аутентифікується (отримує JWT-токен), а далі при кожному запиті разом з токеном отримує дозвіл або відмову на основі правил авторизації.

### **JWT та хешування паролів**

**JWT** (JSON Web Token) — компактний токен у форматі `header.payload.signature`. Сервер підписує payload секретним ключем; будь-яка зміна payload зробить підпис недійсним. Це дозволяє довіряти даним усередині токена без збереження сесії на сервері.

**bcrypt** — алгоритм одностороннього хешування паролів зі вбудованою сіллю. Хеш не можна перетворити назад у пароль; перевірка виконується через `bcrypt.compare`, яка порівнює введений пароль з хешем.

```typescript
const hashedPassword = await bcrypt.hash(plainPassword, 10);
const isValid = await bcrypt.compare(plainPassword, hashedPassword);
```

### **Guards та власні декоратори**

**Guard** — клас, що реалізує метод `canActivate`. Виконується між middleware та контролером і вирішує, чи дозволити виконання обробника. У разі відмови повертає `403 Forbidden` або (для аутентифікації) `401 Unauthorized`.

**Власний декоратор параметра** створюється через `createParamDecorator` і дозволяє витягати дані з request у зручному вигляді — наприклад, замість `@Req() req` написати `@CurrentUser() user`.

## **Завдання**

Реалізувати застосунок на Nest.js + TypeORM + PostgreSQL з аутентифікацією за JWT.

### **Сутність User**

| Поле        | Тип            | Примітка                              |
| ----------- | -------------- | ------------------------------------- |
| `id`        | первинний ключ | UUID або auto-increment               |
| `email`     | рядок          | унікальний, обов'язковий              |
| `password`  | рядок          | зберігається хеш, не відкритий пароль |
| `createdAt` | дата           | автоматично при створенні             |

### **Ендпоінти**

#### `POST /auth/register` — реєстрація користувача

**Доступ:** публічний

**Тіло запиту:**

```json
{
  "email": "user@example.com",
  "password": "secret12345"
}
```

> Конкретні правила валідації полів (мінімальна довжина пароля, формат email тощо) визначаєте самостійно через декоратори `class-validator` у DTO.

**Відповідь (201 Created):**

```json
{
  "id": "<id користувача>",
  "email": "user@example.com",
  "createdAt": "2025-05-15T10:30:00.000Z"
}
```

#### `POST /auth/login` — логін

**Доступ:** публічний

**Тіло запиту:**

```json
{
  "email": "user@example.com",
  "password": "secret12345"
}
```

**Відповідь (200 OK):**

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}
```

#### `GET /auth/me` — поточний користувач

**Доступ:** JWT (заголовок `Authorization: Bearer <token>`)

**Тіло запиту:** відсутнє

**Відповідь (200 OK):**

```json
{
  "id": "<id користувача>",
  "email": "user@example.com"
}
```

### **JWT payload**

При підписанні токена сервер кладе у payload щонайменше:

```json
{
  "sub": "<id користувача>",
  "email": "user@example.com",
  "iat": 1747305000,
  "exp": 1747308600
}
```

Поля `iat` і `exp` додає бібліотека `@nestjs/jwt` автоматично на основі налаштувань модуля. Секрет та час життя токена читаються з `.env` через `ConfigService`:

```
JWT_SECRET=<довгий випадковий рядок>
JWT_EXPIRES_IN=1h
```

> Згенеруйте власний секрет, наприклад через `openssl rand -base64 64`. Не використовуйте короткі або передбачувані значення.

### **Власний декоратор**

| Декоратор        | Тип      | Призначення                                                              |
| ---------------- | -------- | ------------------------------------------------------------------------ |
| `@CurrentUser()` | Параметр | Витягує об'єкт користувача з `request.user` у параметр методу контролера |

Створюється через `createParamDecorator` і використовується замість `@Req() req` + `req.user`.

### **Очікувана поведінка**

- Пароль користувача **ніколи** не повертається у відповідях API (ні в `register`, ні в `me`)
- При некоректних вхідних даних (невалідний email, занадто короткий пароль тощо) → `400 Bad Request`
- При реєстрації з уже існуючим email → `409 Conflict`
- При невірних облікових даних на `POST /auth/login` → `401 Unauthorized`
- При запиті `GET /auth/me` без токена або з невалідним/простроченим токеном → `401 Unauthorized`

### **Архітектурні вимоги**

- Окремі модулі: `AuthModule` та `UsersModule`
- `AuthService` використовує `UsersService`, але не реалізує логіку зберігання користувачів самостійно
- `POST /auth/login` реалізується у `AuthService`: пошук користувача за email, перевірка пароля через `bcrypt.compare`. У разі відсутності користувача або невірного пароля — `UnauthorizedException` (з однаковим повідомленням, щоб не розкривати, чи існує користувач з таким email). У разі успіху — підпис JWT та повернення `{ access_token }`
- `JwtStrategy` (Passport-стратегія) витягує токен із заголовка `Authorization: Bearer <token>`, перевіряє підпис ключем з `JWT_SECRET`. Метод `validate(payload)` формує об'єкт користувача з полів payload (наприклад, `{ id: payload.sub, email: payload.email }`) — цей об'єкт Passport покладе у `request.user`
- `JwtAuthGuard` — власний клас, обгортка над `AuthGuard('jwt')`
- DTO для реєстрації та логіна з валідацією через `class-validator`

## **Типові помилки та їх вирішення**

| Проблема                                          | Рішення                                                                                                                                                   |
| ------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Unauthorized` при коректному токені              | Перевірте, що `JwtStrategy` використовує той самий `JWT_SECRET`, що й `JwtModule.registerAsync`                                                           |
| `req.user` дорівнює `undefined` у контролері      | Переконайтесь, що `@UseGuards(JwtAuthGuard)` застосований; Passport заповнює `req.user` саме з результату `JwtStrategy.validate`                          |
| Пароль повертається у відповіді API               | Прибирайте поле `password` через деструктуризацію `const { password, ...rest } = user` перед поверненням, або використайте `@Exclude` з class-transformer |
| `secretOrPrivateKey must have a value` при логіні | `JwtModule` не підхопив секрет. Використовуйте `registerAsync` з ін'єкцією `ConfigService`, а не `register({ secret: process.env.JWT_SECRET })`           |
| Логін повертає 200 при невірному паролі           | Переконайтесь, що `bcrypt.compare` викликаний з `await` — без `await` функція повертає `Promise`, який є truthy                                           |
