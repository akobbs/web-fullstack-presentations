# **Лабораторна робота №11**

**Тема:** Завантаження файлів

**Мета:** Реалізувати повноцінний застосунок для завантаження файлів з frontend та backend частинами. На стороні Nest.js освоїти прийом файлів через Multer, валідацію типу й розміру та безпечне зберігання. На стороні React освоїти формування `multipart/form-data` через `FormData`, попередній перегляд обраного файлу та відстеження прогресу завантаження.

**Обладнання:** IBM-сумісні ПК

**Програмне забезпечення:** ОС Windows 10/11, macOS або Linux; Node.js (v18+); npm; веб-браузер з інструментами розробника; VS Code; обліковий запис GitHub; Git; HTTP-клієнт для перевірки API (curl, Postman або Bruno)

---

## **Контрольні запитання**

1. Чому для передачі файлів використовують формат `multipart/form-data`, а не `application/json`?
2. Що таке boundary у multipart-запиті та чому заголовок `Content-Type` не можна задавати вручну при відправленні `FormData`?
3. Як на клієнті зробити попередній перегляд зображення до його відправлення і навіщо потрібно звільняти створений об'єктний URL?
4. Як на клієнті відстежити прогрес завантаження файлу на сервер?
5. Чому не можна довіряти заявленому `mimetype` та розширенню файлу і які кроки роблять зберігання завантажених файлів безпечним?

## **Теоретичні відомості**

### **Формат multipart/form-data**

Файли є бінарними даними, тобто послідовністю байтів, яку не можна напряму вкласти у текстовий JSON. Стандартом для завантаження файлів у вебі є формат `multipart/form-data`. Він ділить тіло запиту на окремі частини за допомогою унікального рядка-роздільника (boundary), де кожна частина має власні заголовки та може містити як текстове поле, так і бінарний вміст файлу.

### **FormData та прогрес завантаження**

`FormData` є вбудованим у браузер інтерфейсом для формування набору пар ключ-значення, що автоматично кодується у `multipart/form-data`. Браузер сам генерує boundary та виставляє коректний заголовок `Content-Type`, тому задавати його вручну не можна. HTTP-клієнт `axios` під час передачі тіла викликає колбек `onUploadProgress`, який повідомляє, скільки байтів уже надіслано, що дозволяє обчислити відсоток та відобразити смугу прогресу.

### **Попередній перегляд файлу**

Щоб показати прев'ю зображення до відправлення, об'єкт `File` перетворюють на URL через `URL.createObjectURL`. Це посилання вказує на файл у пам'яті браузера, тому після завершення перегляду його слід звільняти через `URL.revokeObjectURL`, інакше виникає витік пам'яті. Альтернативою є `FileReader`, що читає вміст у формат data URL (Base64).

### **Прийом файлів у Nest.js**

Nest.js обробляє завантаження через бібліотеку Multer з пакета `@nestjs/platform-express`. На метод контролера навішують інтерсептор `FileInterceptor` із зазначенням назви поля, а сам файл отримують через декоратор `@UploadedFile`. Файл надається як об'єкт із метаданими (`originalname`, `mimetype`, `size`, `buffer` або `path`). Назва поля у `FileInterceptor` має точно збігатися з назвою, що надсилає клієнт.

### **Валідація та безпека**

Завантаження файлів є небезпечним вектором атак, оскільки клієнт керує і вмістом, і назвою, і заявленим типом файлу. Nest.js надає `ParseFilePipe` з валідаторами розміру (`MaxFileSizeValidator`) та типу (`FileTypeValidator`). Заявлений `mimetype` та розширення можна підробити, тому надійна перевірка спирається на сигнатуру файлу (magic numbers). Для безпечного зберігання файлу генерують власну унікальну назву (наприклад через UUID), що унеможливлює path traversal та перезапис, а також задають жорсткі ліміти розміру.

## **Завдання**

Реалізувати застосунок для завантаження зображень, що складається з backend-сервісу на Nest.js та frontend-клієнта на React. Проєкт є самодостатнім і не залежить від попередніх лабораторних робіт.

Рекомендована структура репозиторію:

```
lab11/
├── backend/    (Nest.js + Multer)
└── frontend/   (Vite + React + TypeScript + axios)
```

### **Backend (Nest.js)**

Реалізувати сервіс прийому, зберігання та роздачі завантажених файлів. Сервер повинен надавати такі ендпоінти:

| Метод  | Маршрут        | Призначення                                                                   |
| ------ | -------------- | ----------------------------------------------------------------------------- |
| `POST` | `/files`       | Прийняти один файл з поля `file`, провалідувати, зберегти, повернути метадані |
| `GET`  | `/files`       | Повернути список метаданих усіх завантажених файлів                           |
| `GET`  | `/files/:name` | Повернути (роздати) збережений файл за його назвою на сервері                 |

Вимоги до backend:

- Прийом файлу реалізований через `FileInterceptor` та декоратор `@UploadedFile`. Назва поля у клієнта та сервера узгоджена (`file`)
- Валідація через `ParseFilePipe` відхиляє файл, що не відповідає умовам: дозволені лише зображення (`image/jpeg`, `image/png`, `image/webp`), максимальний розмір 5 МБ
- Жорсткий ліміт розміру заданий також на рівні Multer через опцію `limits.fileSize` у `FileInterceptor`. Це обриває прийом завеликого файлу рано, ще до повного завантаження у пам'ять, тоді як `ParseFilePipe` спрацьовує вже після прийому файлу
- Файл зберігається у файловій системі під власною згенерованою унікальною назвою (наприклад UUID плюс розширення), а не під `originalname`
- Метадані завантажених файлів (згенерована назва, початкова назва, розмір, MIME-тип, URL для доступу) зберігаються у пам'яті сервісу. Використання бази даних у цій роботі не потрібне
- Успішна відповідь на `POST /files` має статус `201` і містить метадані збереженого файлу разом з URL, за яким файл доступний
- Ендпоінт `GET /files/:name` повертає вміст файлу з коректним типом

### **Frontend (React)**

Реалізувати односторінковий клієнт, що дозволяє обрати зображення, переглянути його до відправлення, завантажити на сервер з відображенням прогресу та побачити результат.

Очікувана поведінка:

- Користувач обирає файл через елемент вибору файлу (приймаються лише зображення)
- Після вибору відображається попередній перегляд (прев'ю) обраного зображення та його метадані: назва та розмір
- Перед відправленням виконується клієнтська валідація типу та розміру; за невідповідності виводиться зрозуміле повідомлення, а відправлення блокується
- Під час завантаження відображається смуга прогресу з відсотком надісланих байтів, а кнопка відправлення заблокована
- Після успішного завантаження виводиться повідомлення про успіх та відображається зображення, що повернув сервер (за URL з відповіді)
- Помилки сервера (відхилення за типом або розміром) відображаються користувачу в зрозумілому вигляді
- При зміні обраного файлу або після завершення роботи з прев'ю раніше створений об'єктний URL звільняється

Архітектурні вимоги:

- Запит формується через об'єкт `FormData`; заголовок `Content-Type` вручну не встановлюється
- Відправлення виконується через `axios`; прогрес відстежується через `onUploadProgress`
- Прев'ю реалізоване через `URL.createObjectURL` зі звільненням ресурсу через `URL.revokeObjectURL` у функції очищення `useEffect`
- URL бекенду читається зі змінної середовища `VITE_API_URL` (файл `.env`)

### **CORS**

Для роботи frontend-застосунку з backend необхідно увімкнути CORS на стороні Nest.js. У файлі `main.ts` додайте:

```ts
app.enableCors({
  origin: "http://localhost:5173",
});
```

## **Типові помилки та їх вирішення**

| Проблема                                                         | Рішення                                                                                                                                                            |
| ---------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Файл не приходить на сервер, поле порожнє                        | Назва поля у `FormData.append("file", ...)` має збігатися з назвою у `FileInterceptor("file")`                                                                     |
| `multipart: Boundary not found` або тіло не розбирається         | Не встановлюйте заголовок `Content-Type` вручну. axios виставляє його з коректним boundary самостійно для `FormData`                                               |
| Тип `Express.Multer.File` не розпізнається у TypeScript          | Встановіть типи Multer: `npm i -D @types/multer`                                                                                                                   |
| Прев'ю не оновлюється при виборі іншого файлу                    | Зберігайте об'єктний URL у стані та перестворюйте його при зміні файлу; старий URL звільняйте через `revokeObjectURL`                                              |
| Смуга прогресу «застигає» на 100% до отримання відповіді         | Це очікувано: прогрес стосується відправлення, а валідація й зберігання на сервері відбуваються вже після передачі                                                 |
| `CORS policy: No 'Access-Control-Allow-Origin' header` у консолі | Увімкніть CORS на бекенді через `app.enableCors`. Перевірте, що origin збігається з адресою dev-сервера frontend                                                   |
| Завеликий файл повертає помилку без зрозумілого повідомлення     | Обробляйте відповідь сервера на клієнті та виводьте повідомлення; ліміт розміру задавайте у `ParseFilePipe`                                                        |
| Сервер споживає забагато пам'яті при великих завантаженнях       | З memoryStorage файл повністю тримається у RAM. Задавайте `limits.fileSize` у `FileInterceptor`, щоб обірвати прийом завеликого файлу до його повного завантаження |
