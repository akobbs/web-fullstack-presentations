## **Теоретичні відомості**

### **Контейнеризація та образи**

**Контейнер** це ізольований процес, що виконується на хостовій ОС із власною файловою системою, мережею та набором залежностей. На відміну від віртуальної машини, контейнер не містить окремої гостьової операційної системи, а використовує ядро хоста, тому є значно легшим та запускається за секунди.

**Образ (image)** це незмінний шаблон, з якого створюється контейнер. Образ складається з шарів (layers): кожна інструкція у Dockerfile додає новий шар. **Контейнер** це запущений екземпляр образу. З одного образу можна створити багато контейнерів.

Ключова перевага контейнеризації: застосунок працює однаково на машині розробника, на сервері колеги та у продакшені, оскільки разом із кодом упаковується все оточення.

### **Dockerfile**

**Dockerfile** це текстовий файл з інструкціями для збірки образу. Основні інструкції:

- `FROM` базовий образ, від якого успадковується збірка
- `WORKDIR` робоча директорія всередині образу
- `COPY` копіювання файлів з контексту збірки в образ
- `RUN` виконання команди під час збірки (наприклад, встановлення залежностей)
- `EXPOSE` декларація порту, який слухає застосунок (інформативна)
- `CMD` команда, що виконується під час запуску контейнера

```dockerfile
FROM node:lts-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["node", "dist/main.js"]
```

### **Багатоетапна збірка (Multi-stage build)**

Багатоетапна збірка дозволяє використати один образ для компіляції застосунку, а інший, мінімальний, для його виконання. Це суттєво зменшує розмір фінального образу: інструменти збірки та проміжні файли залишаються на етапі build і не потрапляють до результату.

```dockerfile
# Етап збірки
FROM node:lts-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Етап виконання
FROM node:lts-alpine AS production
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=build /app/dist ./dist
CMD ["node", "dist/main.js"]
```

Інструкція `COPY --from=build` переносить лише результат компіляції з попереднього етапу, а директива `--omit=dev` встановлює тільки production-залежності.

### **`.dockerignore`**

Файл `.dockerignore` визначає, що не слід передавати у контекст збірки. Виключення `node_modules`, локальних `.env`, директорії `.git` та артефактів збірки прискорює процес та зменшує розмір образу. Локальні `node_modules` особливо важливо виключати, оскільки залежності повинні встановлюватися всередині образу під його платформу.

```
node_modules
dist
.git
.env
*.log
```

### **Образ для frontend та Nginx**

Зібраний React-застосунок це набір статичних файлів (HTML, CSS, JavaScript). Для їх роздачі не потрібен Node.js: достатньо легкого веб-сервера. Стандартним рішенням є **Nginx**, який віддає статику та може проксіювати запити до backend.

```dockerfile
FROM node:lts-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM nginx:alpine AS production
COPY --from=build /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

### **Docker Compose**

**Docker Compose** це інструмент для опису та запуску багатоконтейнерних застосунків. Усі сервіси, мережі та томи описуються декларативно в одному файлі `compose.yaml`. Команда `docker compose up` запускає весь застосунок однією дією.

Сервіси, описані в одному файлі, автоматично потрапляють у спільну мережу та звертаються один до одного **за іменем сервісу** замість IP-адреси. Наприклад, backend підключається до бази даних за хостом `db`, а не `localhost`.

```yaml
services:
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - postgres_data:/var/lib/postgresql/data

  backend:
    build: ./backend
    depends_on:
      - db
    environment:
      DB_HOST: db

volumes:
  postgres_data:
```

### **Томи та персистентність даних**

Файлова система контейнера є тимчасовою: після його видалення всі зміни втрачаються. Для збереження даних бази між перезапусками використовують **іменований том (named volume)**, керований Docker. На відміну від bind mount, що монтує конкретну директорію хоста, named volume не залежить від структури файлової системи хоста та є переносимим між середовищами.

### **Змінні середовища та порядок запуску**

Чутливі дані (паролі, секрети) не записують у Dockerfile чи `compose.yaml` напряму. Їх передають через файл `.env`, який Docker Compose підставляє у синтаксис `${ЗМІННА}`. Сам файл `.env` не потрапляє у систему контролю версій.

Директива `depends_on` визначає порядок **старту** контейнерів, але не гарантує готовності сервісу приймати запити. Для перевірки фактичної готовності бази даних застосовують `healthcheck` у поєднанні з умовою `depends_on: condition: service_healthy`.

## **Хід роботи**

У цій лабораторній роботі ви контейнеризуєте повний застосунок, що складається з трьох сервісів: backend на Nest.js, frontend на React та база даних PostgreSQL. Усі сервіси будуть запускатися однією командою через Docker Compose.

### **Етап 1: Підготовка репозиторію та структури проєкту**

1.1. Використайте наявний репозиторій курсу `web-programming-labs`. Перейдіть у локальну папку репозиторію.

1.2. Створіть папку `lab12` з такою структурою:

```
lab12/
├── backend/          # Nest.js застосунок з попередніх лабораторних
├── frontend/         # React застосунок з попередніх лабораторних
├── compose.yaml
├── .env.example
└── README.md
```

1.3. Скопіюйте у папки `backend` та `frontend` робочі версії застосунків, створених у попередніх лабораторних роботах. Backend має містити підключення до PostgreSQL через TypeORM, frontend має взаємодіяти з API backend. Переконайтесь, що обидва застосунки запускаються локально без помилок.

> **Примітка:** якщо власні застосунки відсутні або неповні, узгодьте з викладачем мінімальний варіант: backend з одним CRUD-ресурсом та підключенням до бази, frontend з відображенням даних цього ресурсу.

1.4. Зафіксуйте початкову структуру:

```bash
git add lab12/
git commit -m "Lab 12: Initial project structure"
```

**Результат:** репозиторій містить папку `lab12` з підготовленими застосунками.

### **Етап 2: Контейнеризація backend**

2.1. У папці `backend` створіть файл `.dockerignore`, що виключає `node_modules`, `dist`, локальні `.env`-файли, директорію `.git` та лог-файли.

2.2. Створіть `backend/Dockerfile` з **багатоетапною збіркою**, що:

- На етапі `build` встановлює всі залежності та компілює застосунок (`npm run build`)
- На етапі `production` встановлює лише production-залежності (`npm ci --omit=dev`) та копіює результат компіляції з етапу build
- Використовує базовий образ `node:lts-alpine`
- Декларує порт застосунку через `EXPOSE`
- Запускає скомпільований застосунок командою `CMD`

  2.3. Переконайтесь, що backend читає параметри підключення до бази даних (хост, порт, користувач, пароль, назва бази) **зі змінних середовища**, а не з захардкоджених значень. Хост бази має бути конфігурованим, оскільки в контейнерному середовищі він відрізнятиметься від `localhost`.

  2.4. Зберіть образ backend локально та переконайтесь у відсутності помилок збірки:

```bash
docker build -t lab12-backend ./backend
```

2.5. Зафіксуйте зміни:

```bash
git add .
git commit -m "Lab 12: Add backend Dockerfile with multi-stage build"
```

**Результат:** образ backend успішно збирається, інструменти збірки не потрапляють у фінальний образ.

### **Етап 3: Контейнеризація frontend**

3.1. У папці `frontend` створіть файл `.dockerignore` з аналогічними виключеннями.

3.2. Створіть файл `frontend/nginx.conf` з конфігурацією Nginx, що:

- Роздає статичні файли застосунку
- Коректно обробляє маршрути односторінкового застосунку (SPA): будь-який невідомий шлях повертає `index.html`, щоб клієнтська маршрутизація працювала після перезавантаження сторінки
- Проксіює запити з префіксом `/api` до сервісу backend за його іменем у мережі Compose

> **Підказка щодо SPA:** для коректної роботи клієнтської маршрутизації директива має вигляд `try_files $uri $uri/ /index.html;`. Без неї пряме звернення до маршруту, наприклад `/tasks`, поверне помилку 404.

3.3. Створіть `frontend/Dockerfile` з багатоетапною збіркою, що:

- На етапі `build` компілює React-застосунок (`npm run build`)
- На етапі `production` використовує образ `nginx:alpine`, копіює зібрану статику у директорію Nginx та підключає створену конфігурацію

  3.4. Переконайтесь, що адреса API у frontend є конфігурованою (через змінну середовища збірки або через відносний шлях `/api`, що проксіюється Nginx), а не вказана як `http://localhost:3000` жорстко.

  3.5. Зберіть образ frontend локально:

```bash
docker build -t lab12-frontend ./frontend
```

3.6. Зафіксуйте зміни:

```bash
git add .
git commit -m "Lab 12: Add frontend Dockerfile with Nginx"
```

**Результат:** образ frontend збирається та містить лише статику й веб-сервер.

### **Етап 4: Опис середовища через Docker Compose**

4.1. У папці `lab12` створіть файл `.env.example` з переліком необхідних змінних **без реальних значень** (порожні або з плейсхолдерами):

```
DB_USER=app_user
DB_PASSWORD=change_me
DB_NAME=app_db
```

4.2. Створіть локальний файл `.env` з реальними значеннями на основі `.env.example`. Додайте `.env` до `.gitignore`, щоб він не потрапив у репозиторій. Файл `.env.example` навпаки **залишається** у репозиторії як документація.

4.3. Створіть файл `lab12/compose.yaml`, що описує три сервіси:

**Сервіс `db`:**

- Базується на образі `postgres:16-alpine`
- Отримує користувача, пароль та назву бази зі змінних середовища
- Зберігає дані у **іменованому томі** (named volume), щоб вони не втрачалися при перезапуску
- Має `healthcheck`, що перевіряє готовність бази приймати з'єднання (команда `pg_isready`)

**Сервіс `backend`:**

- Збирається з папки `./backend`
- Залежить від `db` з умовою готовності: `depends_on: db: condition: service_healthy`
- Отримує параметри підключення до бази через змінні середовища, де хост бази дорівнює імені сервісу `db`
- Публікує свій порт назовні для перевірки

**Сервіс `frontend`:**

- Збирається з папки `./frontend`
- Залежить від `backend`
- Публікує порт `80` контейнера на доступний порт хоста (наприклад, `8080:80`)

Опишіть також **іменований том** для даних PostgreSQL у секції `volumes` верхнього рівня.

> **Зверніть увагу:** у Docker Compose v2 поле `version:` на початку файлу більше не використовується та вважається застарілим.

4.4. Зафіксуйте зміни:

```bash
git add .
git commit -m "Lab 12: Add Docker Compose configuration"
```

**Результат:** усе середовище застосунку описане декларативно в одному файлі.

### **Етап 5: Запуск та перевірка повного застосунку**

5.1. Запустіть весь застосунок однією командою:

```bash
docker compose up --build
```

5.2. Дочекайтесь, доки всі три контейнери стартують. У логах backend має бути видно успішне підключення до бази даних. Перевірте стан контейнерів:

```bash
docker compose ps
```

5.3. Відкрийте frontend у браузері за адресою хоста (наприклад, `http://localhost:8080`). Переконайтесь, що:

- Сторінка завантажується
- Дані з backend відображаються коректно
- Операції зі створення та читання даних працюють (запити проходять через проксі до backend, а backend звертається до бази)

  5.4. Перевірте **персистентність даних**: створіть запис через інтерфейс, потім перезапустіть застосунок та переконайтесь, що дані збереглися:

```bash
docker compose down
docker compose up
```

> **Примітка:** `docker compose down` без прапора `-v` зупиняє та видаляє контейнери, але **зберігає** іменовані томи. Дані PostgreSQL мають залишитися на місці.

5.5. Зупиніть застосунок коректно:

```bash
docker compose down
```

**Результат:** повний застосунок запускається однією командою, сервіси взаємодіють через спільну мережу, дані бази зберігаються між перезапусками.

| Backend не підключається до бази: `ECONNREFUSED 127.0.0.1`      | Хост бази захардкоджено як `localhost`. У середовищі Compose хостом має бути ім'я сервісу `db`                     |
| Backend стартує раніше за базу та падає з помилкою з'єднання    | Додайте `healthcheck` для `db` та умову `depends_on: condition: service_healthy` для backend                       |
| Образ занадто великий (понад 1 ГБ)                              | Перевірте `.dockerignore` (має виключати `node_modules`) та використання багатоетапної збірки з `--omit=dev`       |
| Frontend повертає 404 при перезавантаженні сторінки на маршруті | У `nginx.conf` додайте `try_files $uri $uri/ /index.html;` для коректної роботи SPA-маршрутизації                  |
| Запити frontend до API не доходять до backend                   | Перевірте налаштування проксі `/api` у `nginx.conf` та що адреса API не вказана як `localhost` жорстко             |
| Дані бази зникають після `docker compose down`                  | Переконайтесь, що використовується іменований том, та не застосовуйте прапор `-v`, який видаляє томи               |
| `.env` потрапив у репозиторій                                   | Додайте `.env` до `.gitignore` до першого коміту; для вже відстежуваного файлу використайте `git rm --cached .env` |
| Зміни у коді не відображаються після `docker compose up`        | Перебудуйте образ з прапором `--build`, оскільки Compose повторно використовує наявний образ за замовчуванням      |
