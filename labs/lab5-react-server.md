## **Теоретичні відомості**

### **JSON Server**

У цій роботі як бекенд використовується **JSON Server** — інструмент, що підіймає повноцінний REST API на основі звичайного JSON-файлу (`db.json`). На відміну від публічних тестових API, JSON Server реально зберігає зміни: після POST новий запис залишається в списку, після DELETE — зникає назавжди.

Сервер запускається локально і доступний за адресою `http://localhost:3001`.

Ресурс, з яким працює ця лаба — `/todos`. Кожен об'єкт має поля: `id`, `title`, `completed`.

### **HTTP-методи та REST**

REST API відображає операції над ресурсами на HTTP-методи:

| Дія          | Метод    | URL           |
| ------------ | -------- | ------------- |
| Отримати всі | `GET`    | `/todos`      |
| Створити     | `POST`   | `/todos`      |
| Оновити      | `PATCH`  | `/todos/{id}` |
| Видалити     | `DELETE` | `/todos/{id}` |

> `PATCH` та `PUT` — обидва оновлюють ресурс, але по-різному. `PUT` **замінює** об'єкт цілком — якщо надіслати лише `{ completed: true }`, решта полів зникне. `PATCH` **зливає** передані поля з існуючим об'єктом. Для часткових оновлень (наприклад, зміна лише `completed`) завжди використовуйте `PATCH`.

### **fetch vs axios**

Обидва інструменти виконують HTTP-запити і повертають Promise. Головна практична відмінність: `fetch` вважає запит успішним навіть при статусах 4xx/5xx — відповідальність за перевірку лежить на розробнику (`response.ok`). `axios` кидає виключення автоматично при таких статусах.

Для роботи з JSON через `fetch` потрібно явно викликати `.json()` на об'єкті відповіді. `axios` десеріалізує JSON автоматично і повертає дані у полі `response.data`.

Обидва підходи рівноцінні для цієї лаби — оберіть той, що здається зручнішим.

### **Три стани HTTP-запиту**

Будь-який запит до сервера проходить через три стани, і UI має коректно обробляти кожен з них:

- **loading** — запит виконується, дані ще не отримані
- **error** — запит завершився помилкою
- **success** — дані отримані та готові до відображення

Якщо не обробити стан `loading` — користувач побачить порожній список до завершення запиту. Якщо не обробити `error` — застосунок мовчки «зламається» без жодного повідомлення.

### **TanStack Query**

TanStack Query — бібліотека для управління серверним станом. Вона автоматично відстежує стани запитів, кешує отримані дані та дедуплікує однакові запити від різних компонентів.

**Підключення.** Увесь застосунок має бути обгорнутий у `QueryClientProvider` з екземпляром `QueryClient`. Це робиться один раз у `main.tsx`.

**`useQuery`** призначений для отримання даних. Хук приймає об'єкт конфігурації з двома обов'язковими полями: `queryKey` — унікальний масив-ідентифікатор запису в кеші, та `queryFn` — функція, що повертає Promise з даними. Хук повертає об'єкт зі станами `isPending`, `isLoading`, `isFetching`, `isError`, `error` та `data`. `isPending` означає, що даних ще немає (перший запит або після скидання кешу). `isFetching` — запит виконується прямо зараз (включаючи фонові рефетчі). `isLoading` — комбінація обох: перший запит, який ще виконується (`isPending && isFetching`).

**`useMutation`** призначений для операцій запису. На відміну від `useQuery`, він не виконується автоматично — потрібен явний виклик `.mutate()`. Конфігурація приймає `mutationFn` та колбеки `onSuccess`, `onError`. Під час виконання мутації доступне поле `isPending`.

**Інвалідація кешу.** Після успішної мутації (створення, оновлення, видалення) кеш TanStack Query містить застарілі дані. Щоб змусити бібліотеку повторно завантажити список, потрібно викликати `queryClient.invalidateQueries` з відповідним `queryKey`. Екземпляр `queryClient` отримується через хук `useQueryClient`.

### **Етап 1: Створення React-проєкту**

1.1. Перейдіть у локальну папку репозиторію курсу та створіть новий Vite + React + TypeScript проєкт у папці `lab5`:

```bash
npm create vite@latest lab5 -- --template react-ts
cd lab5
npm install
```

1.2. Встановіть TanStack Query (`@tanstack/react-query`) та, якщо обрали axios, встановіть його також.

1.3. Підключіть `QueryClientProvider` у `src/main.tsx` — оберніть ним компонент `<App />`:

```tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query'

const queryClient = new QueryClient()

// всередині render:
<QueryClientProvider client={queryClient}>
  <App />
</QueryClientProvider>
```

Документація: https://tanstack.com/query/latest/docs/framework/react/quick-start

1.4. Видаліть шаблонний вміст `App.tsx` та `App.css`.

1.5. Зафіксуйте початковий стан:

```bash
git add .
git commit -m "Lab 5: Initial Vite + React + TypeScript setup"
git push origin main
```

1.6. Запустіть dev-сервер і переконайтесь, що застосунок завантажується без помилок у консолі:

```bash

```

### **Етап 2: Налаштування JSON Server**

2.1. Встановіть JSON Server у папці `lab5`:

```bash
npm install json-server@0.17
```

2.2. Створіть файл `db.json` з початковими даними у корені папки `lab5`:

```json
{
  "todos": [
    { "id": 1, "title": "Вивчити useQuery", "completed": false },
    { "id": 2, "title": "Реалізувати useMutation", "completed": false },
    { "id": 3, "title": "Налаштувати інвалідацію кешу", "completed": true }
  ]
}
```

2.3. Додайте скрипт запуску сервера до секції `scripts` у `package.json`:

```json
"server": "json-server db.json --port 3001"
```

2.4. JSON Server зберігає зміни безпосередньо у `db.json`. Оскільки цей файл знаходиться в корені проєкту, Vite буде реагувати на його зміни і перезавантажувати React-застосунок після кожного POST/PATCH/DELETE. Щоб уникнути цього, додайте `db.json` до ігнорованих файлів у `vite.config.ts`:

```ts
export default defineConfig({
  plugins: [react()],
  server: {
    watch: {
      ignored: ["**/db.json"],
    },
  },
});
```

2.5. Запустіть JSON Server у **окремому терміналі** і переконайтесь, що він працює:

```bash
npm run server
```

Відкрийте у браузері `http://localhost:3001/todos` — має відображатись масив з трьома об'єктами.

2.6. Зафіксуйте:

```bash
git add .
git commit -m "Lab 5: Add JSON Server setup"
```

> З цього моменту для роботи застосунку необхідно мати два запущених процеси одночасно: `npm run server` та `npm run dev` — у двох окремих терміналах.

## **Типові помилки та їх вирішення**

| Проблема                                                        | Рішення                                                                                                                    |
| --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| Помилка про відсутній `QueryClient` при використанні `useQuery` | `<QueryClientProvider>` має обгортати `<App>` у `main.tsx`, а не навпаки                                                   |
| `fetch` не кидає помилку при статусі 404 або 500                | Перевіряйте `response.ok` після кожного виклику `fetch` і кидайте помилку вручну                                           |
| Список не оновлюється після мутації                             | Переконайтесь, що `invalidateQueries` викликається в `onSuccess` з точно таким самим `queryKey`, як у `useQuery`           |
| TypeScript-помилка при типізації аргументу `mutationFn`         | Явно вкажіть тип параметра функції у `mutationFn`                                                                          |
| Застосунок показує помилку одразу після запуску                 | Переконайтесь, що JSON Server запущений (`npm run server`) — React-застосунок не може отримати дані, якщо бекенд не працює |
| TypeScript-помилка `TS1484` при імпорті типів                   | Використовуйте `import type { Todo }` замість `import { Todo }` — Vite вимагає `type`-імпорти для типів та інтерфейсів     |
| React-застосунок перезавантажується після кожної мутації        | Переконайтесь, що `db.json` додано до `server.watch.ignored` у `vite.config.ts`                                            |
